## 5. Insights and Learnings

- **Silent-skip cascades are a defect *amplifier*, not just a defect.** The dry-run found that fixing the surface bug (#5 in the intake — `db_url_pooled` typo) immediately surfaced two more (#7 psycopg3 driver, #8 mutually-exclusive gcloud flags) that had been hiding behind it for months. Walking past one silent-skip without converting it to fail-loud means the next downstream bug is also hidden. Fix the cascade pattern, not just the visible symptom.

- **Real-world bring-up beats fuzzing every time.** Three days of unit-testing the provisioner caught one bug (the SA-not-exist transient). One real dry-run on a clean GCP project found nine. Schedule "fresh-environment dry-runs" as a recurring activity, not just before workshops.

- **Stderr suppression under `set -euo pipefail` is a foot-gun.** `cmd 2>/dev/null | head` swallows the error AND lets `pipefail` kill the script silently. The pattern looks innocuous and is occasionally even idiomatic. The fix is: capture stderr to a tempfile, check `$?` explicitly, relay the captured stderr on failure.

- **GCP eventual-consistency is consistent enough to classify by error signature.** The `retry_eventual_consistency` helper now handles three distinct propagation windows (`API has not been used`, project-level `IAM_PERMISSION_DENIED`, SA-level `IAM_PERMISSION_DENIED`) using stderr-pattern classification. The pattern is reusable; if a fourth lag appears, add the regex.

- **`org-policies list --project` ≠ effective policy.** It only shows policies defined *at* that project. Org-inherited constraints are visible only via `describe`. This is exactly the kind of GCP API quirk that bites templates because the bug doesn't surface when the org has no policy and surfaces silently when it does.

- **"Bootstrap mints values, then user copies them by hand" is the highest-impact failure mode in the entire setup story.** Eliminated 3 of 9 intake findings with one mechanism (#49 / Step 7). When facilitators won't get a SA email or a WIF resource path right under time pressure, don't make them — `gh secret set` is one shell call.

- **Tests should encode the *original bug*, not a generic invariant.** When fixing a silent skip, the test should fail if someone reverts to the bad pattern. When fixing the DRS probe, Test 11 was reshaped from "skip when not enforced" to "apply when describe errors" — that's the regression guard. Generic "the function works on input X" tests don't catch reverts that re-introduce the bug.

- **Cherry-picking proof-of-fix from a downstream fork is a viable upstream-hardening pattern.** The 5-commit bundle in #55 took ~30 minutes to apply (4 modify/delete conflicts on a downstream-only doc file, all resolved with `git rm`). Total LOC: 89 added, 16 deleted. Worth the trouble for a chain of bug fixes that were already debugged elsewhere.

- **Wrapper completion-without-error is not evidence of full iteration.** The CSV roster wrapper "completed without error" while only processing 1 of 3 rows because `set -e` killed it after row 1's silent failure. From the seat: clean exit, blank `roster-output.csv` body, one project in GCP. Always cross-check row count.

- **The `^;^` delimiter for `gcloud --set-env-vars` is the right default.** `^,^` is the gcloud default and breaks on Postgres URLs (commas in passwords). `^@^` is the obvious next choice and breaks on Postgres URLs (`user@host`). `;` is forbidden in URL authority/path components and not used by gcloud as a separator. Document this in the pattern library.

- **Recency shouldn't drive Ready-column ordering.** A ticket becoming unblocked is a precondition for promotion, not a top-of-queue signal. Sort Ready by impact-per-effort, not by "what just got out of the way".
