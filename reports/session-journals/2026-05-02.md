## 5. Insights and Learnings

- **gh api writes error bodies to STDOUT, not just stderr.** A 404 from
  `gh api repos/.../labels/<name>` returns a JSON `{"message":"Not Found",...}`
  on stdout AND exit code 1. The pattern `var=$(gh api ... 2>/dev/null || true)`
  captures the error body into `var`, making `[ -z "$var" ]` false even when
  the call failed. Always check the exit code separately. (Surfaced in #128.)

- **TTY-based context inference doesn't compose with Codespaces.**
  Codespaces postCreateCommand has a TTY despite running as background
  provisioning. The #97 IS_INTERACTIVE guard didn't fire there. Use
  platform-specific env vars (`CODESPACES`, `CI`, `GITHUB_ACTIONS`) for
  hosting-context branching, not TTY heuristics. (Already in
  `Lesson-codespaces-tty-not-reliable-context-signal` memory entity from
  the prior session — reinforced today.)

- **Architectural assumptions need audit when models shift.** The
  framework had a defensible pre-#108 design choice: "NEON_API_KEY and
  NEON_PROJECT_ID are cohort-shared, so don't auto-push them per-row."
  After #108 made `NEON_PROJECT_ID` per-attendee, that conditional in
  Step 7 became wrong. The stale conditional cost a workshop dry-run
  cycle to surface. Audit code paths that branch on assumptions when
  the underlying model changes; don't trust comments that describe the
  old architecture. (#130's root cause.)

- **`gh repo create --template` semantics: only `main` is meaningful.**
  The flag squashes the template's main into a single fresh commit;
  any other branches copied via `--include-all-branches` retain their
  pre-squash history and share zero commits with new main. Default
  behavior (no flag) is what attendees actually need. (#129's fix.)

- **WIF org-trust pattern is one binding for many repos.** Instead of
  per-repo `attribute.repository/<owner>/<repo>` bindings, use
  `attribute.repository_owner/<org>` with the corresponding
  `assertion.repository_owner == '<org>'` provider condition. Single
  binding covers every repo under the org. Backwards-compatible when
  `WIF_ORG_TRUST_PATTERN` is unset. (Established in #107.)

- **Per-attendee Neon project + cohort-shared API key is the right
  partition.** `NEON_API_KEY` stays cohort-shared (one workshop key,
  rotated per cohort). `NEON_PROJECT_ID` is per-attendee (each gets
  their own project's 10-branch quota under a workshop-scoped Neon
  org). The roster CSV's 7-column variant carries the per-row project
  ID. (Established in #108.)

- **Long test suites + reviewer-agent monitors don't compose
  cleanly.** Reviewer agents that run `./scripts/provision-gcp-project.test.sh`
  (3-5 min) routinely hit their monitor's default 5-min timeout. Either
  give the reviewer explicit instructions to skip the long suite (trust
  author's pre-PR run), or arm the monitor with a longer timeout
  upfront.

- **Triaging downstream Claude sessions: separate symptom from
  diagnosis.** Today's downstream session correctly identified that
  `singleSelectOptionId` is the right mutation field for ProjectsV2,
  but wrapped that finding with three wrong diagnoses (newlines in
  `-f query=`, `read:user` scope requirement, GH_TOKEN precedence).
  Course-correcting via a structured prompt that praised the real find
  and named the wrong ones explicitly worked better than rewriting
  their handoff doc.

- **Pre-existing `_refuse_sqlite_in_production` validator (#63/#78) is
  load-bearing.** When downstream feedback proposes "just add a SQLite
  fallback" to make tests easier, reject it explicitly with the
  historical context. The validator's silent-fallback elimination took
  multiple PRs to ship; reverting would undo months of dry-run learning.
